## 1. 產品願景與策略 (Product Vision & Strategy)

### 願景 (Vision)
讓每一位加密貨幣投資者都能輕鬆實踐「低買高賣」的投資紀律，透過智能化的價格監控與決策建議，協助個人投資者在波動劇烈的加密貨幣市場中，做出理性且有紀律的交易決策。

### 使命 (Mission)
我們透過建立一套簡單易用的價格區間監控與交易建議系統，幫助個人投資者：
1. 克服情緒化交易的弱點
2. 自動化執行預先設定的交易策略
3. 在不需要 24/7 盯盤的情況下，捕捉市場機會

### 核心價值主張 (Unique Value Proposition)
**「設定價格區間，讓系統當你的理性守門員」**

我們提供：
- 簡單直觀的價格上下限設定機制
- 智能監控與即時交易建議通知
- 安全的人工確認機制（避免誤操作）
- 同時支援自動策略與手動操作的彈性

## 3. 目標使用者與問題 (Target Users & Problems)

### 主要使用者畫像 (Primary Persona): 理性但忙碌的個人投資者 - "Alex"

#### 是誰:
- **年齡**: 28-45 歲
- **職業**: 上班族、自由工作者、小型企業主
- **投資經驗**: 接觸加密貨幣 1-3 年，有基本的買賣經驗
- **技術能力**: 中等程度，能使用加密貨幣交易所，但不熟悉程式交易
- **投資目標**: 中長期資產增值，不是日內頻繁交易者

#### 核心痛點 (Pain Points):
1. **情緒化交易**: 市場暴漲時 FOMO（怕錯過）追高買入，暴跌時恐慌賣出，事後常常後悔。
2. **無法 24/7 盯盤**: 白天要工作，晚上要休息，常常錯過理想的買賣價格。
3. **缺乏交易紀律**: 即使心中有「低於 X 就買、高於 Y 就賣」的策略，但實際執行時容易動搖。
4. **操作複雜**: 交易所的進階功能（如限價單、止損單）設定複雜，擔心設錯參數。
5. **資訊過載**: 市場資訊、新聞、技術分析太多，不知道該相信哪一個。

#### 期望 (Goals):
- 希望有一個「理性的自己」能在預先設定的價格點提醒自己該行動了。
- 能在不盯盤的情況下，自動捕捉到符合自己策略的買賣時機。
- 保留最終決策權，系統只是輔助而非完全自動化（避免失控的風險）。
- 能清楚看到自己的持倉、歷史交易記錄，並評估策略效果。

### 次要使用者畫像 (Secondary Persona): 保守的加密貨幣新手 - "Linda"

#### 是誰:
- **年齡**: 25-35 歲
- **職業**: 一般上班族
- **投資經驗**: 剛開始接觸加密貨幣 < 1 年
- **技術能力**: 基本電腦操作，對交易所介面不太熟悉
- **投資目標**: 小額嘗試，想要學習但不想承擔太大風險

#### 核心痛點:
- 不知道什麼時候該買、什麼時候該賣。
- 害怕因為不懂而做出錯誤決策。
- 希望有「系統建議」來學習交易邏輯。

## 4. 核心概念與術語定義 (Core Concepts & Terminology)

### 4.1 價格區間策略 (Price Range Strategy)

**定義**: 使用者為特定加密貨幣設定一個「理想買入價格下限」與「理想賣出價格上限」，系統會監控市場價格，當價格觸及這些區間時，提供交易建議。

**核心邏輯**:
```
IF 市場價格 ≤ 買入價格下限 THEN 建議買入
IF 市場價格 ≥ 賣出價格上限 THEN 建議賣出
```

**範例**:
- 使用者設定 BTC 的買入下限為 $60,000，賣出上限為 $70,000
- 當 BTC 價格跌到 $60,000 或以下時，系統發送「建議買入」通知
- 當 BTC 價格漲到 $70,000 或以上時，系統發送「建議賣出」通知

### 4.2 金額上下限 (Price Bounds)

| 術語 | 定義 | 範例 |
|------|------|------|
| **買入價格下限 (Buy Lower Bound)** | 使用者認為的「理想買入價格」或更低。當市場價格低於此值時，系統認為是好的買入機會。 | BTC 目前 $65,000，使用者設定買入下限為 $60,000 |
| **賣出價格上限 (Sell Upper Bound)** | 使用者認為的「理想賣出價格」或更高。當市場價格高於此值時,系統認為是好的賣出機會。 | BTC 目前 $65,000，使用者設定賣出上限為 $70,000 |

**重要特性**:
- 價格區間由使用者主觀設定，系統不提供「智能建議」（未來可擴充）
- 支援同時設定多個不同加密貨幣的價格區間
- 支援動態調整（使用者隨時可以修改上下限）

### 4.3 交易決策流程 (Trading Decision Flow)

```
1. 價格監控階段
   └─> 系統持續監控市場即時價格

2. 觸發條件檢測
   └─> 比對價格與使用者設定的上下限

3. 決策建議生成
   └─> 當條件滿足時,生成「建議買入」或「建議賣出」的通知

4. 人工確認階段
   └─> 使用者收到通知,可選擇:
       ├─> 接受建議並執行交易（手動操作）
       ├─> 拒絕建議（忽略）
       └─> 調整策略後再執行

5. 交易執行記錄
   └─> 記錄使用者的實際操作（無論是否依照建議）
```

### 4.4 手動操作 vs 自動決策

| 模式 | 定義 | MVP 階段支援 |
|------|------|--------------|
| **手動交易模式** | 使用者可以隨時手動執行買入或賣出操作，不依賴價格區間策略 | ✅ 是 |
| **半自動決策模式** | 系統監控價格並發送建議，但需要使用者人工確認後才執行交易 | ✅ 是 (MVP 核心) |
| **全自動交易模式** | 系統在條件滿足時自動執行交易，不需要人工介入 | ❌ 否 (未來擴充) |

## 6. 產品路線圖 (Roadmap)

### 近期 (Now - Next 3 Months): MVP 開發與驗證階段

**目標成果**: 完成核心功能 MVP，驗證「價格區間策略 + 通知建議」的產品概念是否能解決目標使用者的痛點。

**核心交付項**:
1. ✅ 價格區間策略的建立、查看、編輯、刪除功能 (Epic 1)
2. ✅ 與 Binance API 整合，實現即時價格監控 (Epic 2 - T06, T07, T08)
3. ✅ 手動交易執行與持倉管理 (Epic 3 - T11, T12, T14)
4. ✅ CLI 介面（使用者可透過終端機操作所有功能）(Epic 4 - T15, T16, T18)
5. ✅ 本地資料儲存（SQLite）(Epic 4 - T15)

**成功標準**:
- 能成功監控至少 3 種主流幣種（BTC, ETH, USDT）
- 價格觸發通知延遲 < 30 秒
- 邀請 10 位個人投資者進行 Alpha 測試，收集回饋

---

### 中期 (Next - 3-6 Months): 產品優化與功能擴充

**目標成果**: 根據 Alpha 測試回饋優化使用者體驗，並擴充進階功能，提升產品留存率。

**核心交付項**:
1. 🔄 多種通知方式支援（Email、Telegram Bot、Line Notify）
2. 🔄 策略效果分析儀表板（顯示每個策略的觸發次數、使用者採納率、假設性收益）
3. 🔄 進階策略設定（如梯級買入：價格每跌 5% 就買一次）
4. 🔄 支援更多交易所（Coinbase, Kraken）
5. 🔄 簡易的 Web UI（不再僅限於 CLI）
6. 🔄 回測功能（使用歷史價格資料測試策略效果）

**成功標準**:
- Beta 使用者 50 人，其中 60% 持續使用超過 2 週
- NPS 分數 ≥ 40
- 使用者平均每週檢視策略或收到通知次數 ≥ 3 次

---

### 遠期 (Later - 6+ Months): 進階功能與商業化探索

**目標成果**: 探索產品的商業化可能性，並針對更專業的使用者提供進階功能。

**核心交付項**:
1. 🔮 全自動交易模式（使用者可選擇完全自動執行，不需人工確認）
2. 🔮 與交易所 API 的完整對接（實際執行交易，而非僅記錄）
3. 🔮 智能策略推薦（基於歷史數據與機器學習，建議最佳價格區間）
4. 🔮 社群功能（使用者可以分享自己的策略，參考其他人的設定）
5. 🔮 行動 App（iOS/Android）
6. 🔮 訂閱制（免費版 vs 進階版功能差異）

**成功標準**:
- 月活躍使用者 (MAU) ≥ 500 人
- 付費轉換率 ≥ 5%（如果採用訂閱制）
- 使用者平均執行的交易中，70% 是依照系統建議執行

## 7. 不做的項目 (Out of Scope)

為了保持專案的專注度，我們在 MVP 與中期階段「刻意不做」以下功能：

### MVP 階段不做：
- ❌ 不支援衍生品交易（期貨、選擇權、槓桿交易）
- ❌ 不提供技術分析指標（如 MACD、RSI、均線）
- ❌ 不支援多帳戶管理（一個使用者只能管理一個投資組合）
- ❌ 不提供智能策略建議（由 AI 或演算法自動推薦價格區間）
- ❌ 不實際對接交易所 API 執行真實交易（僅記錄模擬交易）
- ❌ 不支援法幣出入金管理
- ❌ 不提供稅務報表生成

### 中期階段不做：
- ❌ 不支援社群功能（分享策略、跟單）
- ❌ 不支援程式化交易（讓使用者自己寫策略腳本）
- ❌ 不提供完整的投資組合管理（如跨交易所資產整合）

### 永遠不做：
- 🚫 不提供投資建議或保證收益（避免法律風險）
- 🚫 不代客操作資金（使用者自己掌握資金）
- 🚫 不儲存使用者的交易所 API Key 密碼（如果要對接交易所，使用者需要自行設定）

## 8. 假設與風險 (Assumptions & Risks)

### 主要假設:

1. **使用者假設**: 個人投資者確實有「設定價格區間」的需求，且願意使用第三方工具來輔助決策。
2. **技術假設**: 交易所 API（如 Binance）的即時價格資料是穩定且免費可取得的。
3. **行為假設**: 使用者收到「建議通知」後，有足夠的動機真的去執行交易（而不是忽略通知）。
4. **市場假設**: 加密貨幣市場的波動性會持續存在，使得「低買高賣」策略仍然有效。
5. **商業假設**: 即使 MVP 不收費，使用者仍願意投入時間測試與提供反饋。

### 潛在風險:

| 風險類型 | 描述 | 嚴重性 | 緩解措施 |
|----------|------|--------|----------|
| **市場風險** | 加密貨幣市場進入長期熊市，使用者失去交易興趣 | 高 | - 擴展支援其他資產類別（股票、ETF）<br>- 強調「長期投資紀律」的價值主張 |
| **技術風險** | 交易所 API 限流或突然關閉免費存取 | 中 | - 整合多個交易所 API 作為備援<br>- 提前設計 rate limiting 處理機制 |
| **競爭風險** | 主流交易所（如 Binance）推出類似的價格提醒功能 | 高 | - 強調我們的「跨交易所」整合優勢<br>- 提供更進階的策略設定（而非僅簡單提醒） |
| **使用者體驗風險** | 使用者覺得「人工確認」流程太麻煩，希望完全自動化 | 中 | - 在 Beta 測試中驗證使用者對「控制權」的需求<br>- 中期提供「全自動模式」選項 |
| **法律風險** | 提供交易建議可能觸犯金融監管法規 | 中 | - 明確聲明「系統僅提供資訊，非投資建議」<br>- 諮詢法律顧問確認免責聲明的有效性 |
| **資料安全風險** | 本地儲存的使用者資料（策略、交易記錄）可能被駭客竊取 | 中 | - 本地資料庫加密<br>- 不儲存敏感的 API Key（如果有對接交易所需求，由使用者自行保管） |
| **產品黏著度風險** | 使用者設定策略後忘記系統，不再回來檢查通知 | 高 | - 多元化通知管道（Email、Telegram）<br>- 定期發送「策略效果摘要」報告 |

### 技術選型

| 層級 | 技術選擇 | 理由 |
|------|---------|------|
| 後端語言 | **Go (Golang)** | 高效能、原生支援併發（goroutine）、適合背景服務、部署簡單（單一執行檔） |
| CLI 框架 | **Cobra** | Go 生態系最成熟的 CLI 框架，被 kubectl、Hugo 等知名專案使用 |
| 資料庫 | **SQLite** (MVP) | 輕量、無需額外安裝、適合單使用者場景 |
| ORM | **GORM** 或純 SQL | GORM 提供簡單的資料操作，但保留彈性使用純 SQL 進行複雜查詢 |
| HTTP 客戶端 | **net/http** (標準庫) | 原生支援，效能優異 |
| 並發管理 | **Go routine + Channel** | Go 原生支援，適合價格監控背景服務 |
| 測試框架 | **testing** (標準庫) + **testify** | 標準庫提供基礎，testify 提供更好的斷言 |
| 日誌 | **logrus** 或 **zap** | 結構化日誌，易於追蹤問題 |
